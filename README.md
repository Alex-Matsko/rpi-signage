# RPi Signage

Сервис публикации афиш (изображения и видео) на телевизорах через
Raspberry Pi 2/3/4.

- **Сервер** — Docker-контейнер: веб-интерфейс управления (логин/пароль) +
  API для агентов. Структура по городам, публикация афиш одним шагом
  (несколько файлов + расписание + выбор городов/экранов), сроки истечения,
  дашборд с состоянием каждого экрана в реальном времени.
- **Агент** — systemd-сервис на Raspberry Pi: кеширует контент с сервера
  (SHA-256, докачка), проигрывает через mpv (KMS/DRM, без X-сервера,
  аппаратный декод H.264), автономно работает при потере связи.

Полное ТЗ: [docs/TZ.md](docs/TZ.md).

## Как устроено управление

- **Города** — основа структуры: в каждом городе свои экраны (кассы).
  Афиша назначается на город целиком (включая будущие экраны) или на
  отдельные экраны.
- **Публикация одним шагом**: страница «Опубликовать» — перетащите сразу
  несколько файлов, задайте срок и окно показа, отметьте города — готово.
- **Роли**: администратор управляет всем; менеджер города видит и
  управляет только своим городом.

## Удалённое управление экраном

На странице экрана (даже если Raspberry Pi за NAT — всё работает через
исходящие подключения агента):

- **Что сейчас на экране** — кнопка «Сделать скриншот»: агент снимает
  текущий кадр через mpv и присылает его в интерфейс.
- **Терминал (SSH из браузера)** — полноценный shell устройства прямо на
  странице (xterm.js ⇄ WebSocket ⇄ агент поднимает pty). Не требует
  открытых портов на RPi и проброса SSH.
- **Перезапустить агент** и **Перезагрузить Raspberry Pi** — команды с
  историей выполнения. Перезагрузка разрешена через `sudoers` (ставится
  автоматически в install.sh).

## Локальная веб-панель устройства

На каждом устройстве агент поднимает собственную панель настроек на порту
`8088` (`http://<IP-устройства>:8088`, вход по логину/паролю). Она работает
**даже до привязки к серверу**, поэтому всё настраивается без SSH и без
командной строки:

- **Сервер**: привязать устройство к серверу — ввести адрес и код
  подключения прямо в панели (или сменить сервер позже);
- **Сеть**: текущий IP и интерфейсы, подключение к Wi-Fi (NetworkManager);
- **Звук**: выбор аудиовыхода **HDMI** или **аналоговый 3.5 мм**;
- **Хранилище**: свободное место на диске, список кешированных афиш,
  удаление отдельных файлов и полная очистка кеша;
- **Система**: часовой пояс, имя устройства, смена логина/пароля панели,
  перезапуск агента и перезагрузка.

Адрес панели виден на странице экрана в основном интерфейсе. Пароль по
умолчанию **`admin` / `signage`** (задан заранее) — задайте свой при
установке (`--web-password`) или смените в разделе «Система».

## Запуск сервера

```bash
git clone https://github.com/<you>/rpi-signage.git
cd rpi-signage
ADMIN_PASSWORD='ваш-пароль' docker compose up -d
```

Веб-интерфейс: http://localhost:8080 (логин `admin`, порт меняется
переменной `SIGNAGE_PORT`).

Переменные окружения:

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `ADMIN_USER` / `ADMIN_PASSWORD` | `admin` / — | первый администратор (создаётся при пустой БД) |
| `SIGNAGE_PORT` | `8080` | внешний порт веб-интерфейса |
| `SIGNAGE_DATA_PATH` | `./data` | папка с данными (БД, медиа, скриншоты) |
| `TZ` | `Europe/Moscow` | часовой пояс сервера (должен совпадать с RPi) |
| `SIGNAGE_MAX_UPLOAD_MB` | `1024` | лимит размера загружаемого файла |
| `SIGNAGE_POLL_INTERVAL` | `60` | период опроса сервера агентами, сек |

В продакшене поставьте сервер за reverse-proxy с HTTPS (Caddy, nginx,
Traefik) — токены устройств и cookie не должны ходить открытым текстом.

## Данные и обновление без потери контента

Все данные (БД с экранами и афишами, загруженные медиафайлы, скриншоты,
ключ сессий) хранятся в папке **`./data`** рядом с `docker-compose.yml`
(путь меняется переменной `SIGNAGE_DATA_PATH`). Папка переживает пересборку
образа, `docker compose down` и даже `down -v` — удалить данные можно только
вручную, удалив саму папку.

**Обновляйтесь безопасным скриптом** — он делает бэкап и не трогает данные:

```bash
./scripts/update.sh
```

Эквивалент вручную (данные сохраняются):

```bash
git pull && docker compose up -d --build
```

> ⚠️ **Никогда не используйте `docker compose down -v`** для обновления —
> флаг `-v` удаляет тома. С папкой `./data` это уже не удалит контент, но
> привычка опасная.

**Резервные копии:**

```bash
./scripts/backup.sh                              # ./backups/signage-data-*.tar.gz
./scripts/restore.sh backups/signage-data-*.tar.gz   # восстановить
```

### Данные пропали? Сначала найдите их — они почти наверняка на диске

Папка `./data` относительна каталогу, откуда запускается `docker compose`.
Если вы **клонировали репозиторий заново** в другую папку, перенесли проект
или данные раньше лежали в старом Docker-томе — сервер стартует с пустой
базой, хотя данные ещё есть на диске. Найдите их:

```bash
./scripts/find-data.sh
```

Скрипт покажет все базы (в папках и Docker-томах) и сколько в каждой экранов
и афиш. Нашли нужную — перенесите её в рабочую папку:

```bash
./scripts/migrate-data.sh <имя-тома>        # напр. rpi-signage_signage-data
./scripts/migrate-data.sh /путь/к/старой/data
```

> 💡 **Чтобы это не повторялось:** задайте в `.env` **абсолютный** путь,
> не зависящий от каталога запуска, и всегда обновляйтесь на месте
> (`./scripts/update.sh`), не клонируя проект заново:
> ```
> SIGNAGE_DATA_PATH=/opt/rpi-signage/data
> ```

`./scripts/update.sh` дополнительно **остановится**, если увидит пустую
рабочую папку при наличии данных в Docker-томе — чтобы вы не запустили
пустой сервер поверх потерянных данных.

## Подключение экрана (Raspberry Pi / Intel NUC)

Подробная пошаговая инструкция с проверкой — [docs/INSTALL.md](docs/INSTALL.md).
Коротко:

1. В веб-интерфейсе: **Экраны → Добавить экран** — получите одноразовый
   код подключения вида `AB12-CD34`.
2. На Raspberry Pi OS Lite (Bookworm) или Debian/Ubuntu (NUC):

```bash
curl -fsSL http://ВАШ-СЕРВЕР:8080/install.sh | sudo bash -s -- \
  --server http://ВАШ-СЕРВЕР:8080 --code AB12-CD34
```

Скрипт ставит `mpv`, создаёт пользователя `signage`, регистрирует
устройство и запускает systemd-сервис `signage-agent`. Через минуту экран
появится online на дашборде.

Логи агента: `journalctl -u signage-agent -f`.

## Требования к контенту

- Изображения: JPEG, PNG.
- Видео: **MP4 / H.264 (AVC), до 1920×1080, до 30 fps** — общий знаменатель
  аппаратного декодирования RPi 2/3/4. Несовместимое видео (другой кодек,
  4K, высокий fps) сервер **автоматически транскодирует** ffmpeg-ом в фоне;
  статус виден в медиатеке и на афише.

## Расписания показа

У каждой афиши, помимо дат начала и истечения, задаются:

- ежедневное окно показа (например, 08:00–23:00; окно через полночь
  22:00–06:00 тоже работает);
- дни недели.

Агент применяет расписание по локальным часам устройства, поэтому оно
соблюдается даже без связи с сервером.

## Обновление агентов

Агенты обновляются сами: при каждом опросе агент сравнивает свою версию с
версией `agent.py` на сервере, скачивает новую, проверяет её и
перезапускается через systemd. Достаточно обновить сервер (`git pull &&
docker compose up -d --build`) — экраны подтянут новую версию в течение
минуты.

## Разработка

```bash
# Тесты агента (чистые функции расписания)
pytest agent

# Тесты сервера
cd server && pip install -r requirements.txt pytest httpx && pytest tests

# Агент в dev-режиме (mock-плеер, без RPi и mpv)
python3 agent/agent.py --state-dir /tmp/signage-dev \
  register --server http://localhost:8080 --code КОД
python3 agent/agent.py --state-dir /tmp/signage-dev run --dev --poll-interval 5
```

Структура:

```
server/   FastAPI-приложение (веб-интерфейс + API), тесты
agent/    agent.py (stdlib-only), install.sh, systemd unit
docs/     ТЗ и документация
```

## Как это работает

1. Агент опрашивает `GET /api/agent/manifest` (pull — RPi может быть за NAT).
2. Недостающие файлы скачиваются в кеш с докачкой и проверкой SHA-256;
   плейлист применяется только когда контент готов.
3. mpv работает одним долгоживущим процессом (JSON IPC) — смена афиш без
   чёрного экрана; изображениям задаётся длительность показа, видео играет
   целиком.
4. Каждые 30 секунд (и при каждой смене афиши) агент шлёт heartbeat:
   что играет, кеш, температура CPU, свободное место.
5. Сроки действия афиш агент соблюдает по локальным часам — истёкшее
   пропадает с экрана даже без связи с сервером.
