# Техническое задание: Signage — сервис публикации афиш на ТВ через Raspberry Pi

Версия: 0.1 (черновик на согласование) · Дата: 2026-07-16

## 1. Цель

Система для показа афиш (изображения и видео) на телевизорах, подключённых к
Raspberry Pi 2 / 3 / 4. Состоит из двух компонентов:

- **Сервер** — контейнер Docker: REST API + веб-интерфейс управления.
- **Агент** — сервис на Raspberry Pi: скачивает и кеширует контент с сервера,
  проигрывает его на ТВ, отчитывается о состоянии.

## 2. Общая архитектура

```
┌────────────────────────────┐        HTTPS (pull)        ┌─────────────────┐
│  Сервер (Docker)           │ <────────────────────────  │  Агент на RPi   │
│  FastAPI + SQLite + Web UI │  manifest / файлы / статус │  кеш + плеер    │
│  тома: /data (БД, медиа)   │ ────────────────────────>  │  mpv (KMS/DRM)  │
└────────────────────────────┘                            └───────┬─────────┘
        ▲ браузер (логин/пароль)                                  │ HDMI
        └── администратор                                         ▼ Телевизор
```

Принципы:
- Агент **сам опрашивает** сервер (pull) — работает за NAT, не требует
  проброса портов к RPi.
- Агент полностью автономен при потере связи: играет из локального кеша,
  соблюдая сроки истечения по локальным часам (NTP).
- Весь контент адресуется по SHA-256 — докачка, проверка целостности,
  дедупликация.

## 3. Сервер

### 3.1 Стек

- Python 3.12, FastAPI, Uvicorn.
- БД: SQLite (файл на томе Docker); архитектура через SQLAlchemy —
  миграция на PostgreSQL возможна без переписывания.
- Веб-интерфейс: серверные шаблоны Jinja2 + Alpine.js + htmx
  (без отдельной сборки фронтенда, минимум зависимостей).
- Один образ Docker; docker-compose для локального запуска.
- Тома: `/data/db` (БД), `/data/media` (оригиналы и превью).

### 3.2 Аутентификация и пользователи

- Вход по логину/паролю, сессии (cookie, HttpOnly), пароли — argon2.
- Первый администратор создаётся из переменных окружения
  (`ADMIN_USER` / `ADMIN_PASSWORD`) при первом старте.
- Управление пользователями в UI (создать/удалить, смена пароля).
- Rate-limit на форму входа.

### 3.3 Медиатека

- Загрузка: JPEG, PNG (изображения); MP4 (видео).
- При загрузке: подсчёт SHA-256, определение параметров через ffprobe,
  генерация превью (для видео — кадр).
- **Проверка совместимости:** видео должно быть H.264 (AVC) ≤1080p ≤30fps —
  общий знаменатель аппаратного декодирования RPi 2/3/4. Если кодек другой —
  предупреждение в UI (опционально в фазе 2 — авто-транскодирование ffmpeg).
- Лимит размера файла настраивается (по умолчанию 1 ГБ).

### 3.4 Афиши и плейлисты

- Афиша = медиафайл + параметры:
  - название;
  - длительность показа (для изображений, сек; для видео — длина ролика);
  - дата/время начала показа (опционально);
  - **дата/время истечения** (после — автоматически убирается с экранов);
  - признак активности (ручное вкл/выкл).
- Плейлист = упорядоченный список афиш; крутится по кругу.
- Плейлист назначается на устройство или группу устройств.

### 3.5 Устройства (экраны)

- Регистрация: в UI создаётся устройство → одноразовый код подключения →
  вводится при установке агента → агент получает постоянный токен.
- Группы устройств (например «Холл», «Кассы»).
- Карточка устройства:
  - online/offline (по heartbeat, порог ~90 сек);
  - **что показывается прямо сейчас** (имя афиши + время начала показа);
  - состояние кеша: сколько файлов из манифеста скачано;
  - версия агента, аптайм, температура CPU, свободное место на SD;
  - назначенный плейлист.

### 3.6 Dashboard

- Сводка: все экраны, их статус и текущая афиша.
- Предупреждения: офлайн-экраны, афиши, истекающие в ближайшие N дней,
  устройства с неполным кешем.

### 3.7 API для агента (`/api/agent/*`, авторизация по токену устройства)

- `POST /register` — обмен кода подключения на токен.
- `GET /manifest` — актуальный плейлист устройства: список элементов
  (sha256, url, тип, длительность, срок действия), версия манифеста.
- `GET /media/{sha256}` — скачивание файла; поддержка `Range` (докачка).
- `POST /status` — heartbeat: текущий элемент, метрики, состояние кеша.

## 4. Агент на Raspberry Pi

### 4.1 Требования к платформе

- Raspberry Pi 2, 3, 4; Raspberry Pi OS Lite (Bookworm, 32-bit — единый
  образ для всех трёх поколений).
- Работа без X-сервера: вывод через KMS/DRM.

### 4.2 Стек и устройство

- Python 3.11+ (системный), зависимости минимальны (requests + stdlib).
- Плеер: **mpv** — единый для изображений (`--image-display-duration`)
  и видео (аппаратный декод `v4l2m2m`, вывод `gpu`/`drm`).
- Служба systemd `signage-agent.service`, `Restart=always`; watchdog
  перезапускает mpv при зависании.

### 4.3 Логика работы

1. Каждые `poll_interval` (по умолчанию 60 с) запрашивает манифест.
2. Сравнивает с кешем по SHA-256; недостающее докачивает (с резюмированием),
   проверяет хеш, кладёт в `/var/lib/signage/cache`.
3. Плейлист применяется атомарно: пока новый контент не докачан полностью —
   играет старый.
4. Истекшие элементы исключаются локально по часам устройства (NTP обязателен).
5. Кеш-«сироты» (файлы вне манифеста) удаляются с отсрочкой.
6. Heartbeat каждые 30 с: текущий файл, метрики.
7. Нет сети → играет из кеша неограниченно долго.
8. Пустой плейлист / нет контента → заставка «экран подключён, контент
   не назначен» (генерируется локально).

### 4.4 Установка

- `install.sh`: одна команда с параметрами `SERVER_URL` и код подключения —
  ставит зависимости (mpv), создаёт пользователя, systemd-юнит, конфиг
  `/etc/signage/agent.toml`.
- Обновление агента: скриптом (self-update по команде с сервера — фаза 2).

## 5. Безопасность

- Сервер за reverse-proxy с HTTPS (в docker-compose — опциональный Caddy
  с автосертификатом; можно отключить, если TLS терминируется снаружи).
- Токены устройств — случайные 256-бит, хранятся хешированными.
- Куки-сессии только для UI; API агента — только токен.

## 6. Репозиторий и инфраструктура

- GitHub, монорепозиторий:
  ```
  /server        — FastAPI-приложение, Dockerfile
  /agent         — агент + install.sh
  /docs          — ТЗ, инструкции по развёртыванию
  docker-compose.yml
  ```
- CI (GitHub Actions): линт + pytest + сборка Docker-образа.
- Локальное тестирование: `docker compose up` поднимает сервер;
  у агента есть **dev-режим** (mock-плеер, пишет «что играет» в лог) —
  позволяет проверить весь протокол на Mac/в контейнере без RPi.

## 10. v0.4 — удалённое управление экранами

Всё через pull-модель (агент подключается сам, RPi может быть за NAT):

- ✅ **Скриншот экрана**: команда → агент снимает кадр через mpv
  (`screenshot-to-file`) → загружает на сервер → показывается на странице
  экрана.
- ✅ **Веб-терминал (SSH из браузера)**: страница экрана с xterm.js; браузер
  ⇄ сервер по WebSocket, сервер ⇄ агент по HTTP long-poll, агент поднимает
  интерактивный shell в pty. Не нужны открытые порты и проброс SSH.
- ✅ **Перезапуск агента / перезагрузка RPi**: команды с историей; агент
  выполняет перезагрузку через `sudo reboot` (правило sudoers ставит
  install.sh), опасная команда gated флагом `--allow-system`.
- Инфраструктура команд: таблица `device_commands`, длинный опрос
  `GET /api/agent/commands`, отчёт о результате
  `POST /api/agent/commands/{id}/result`.

## 13. v0.8 — звук через ALSA (headless) и исправления панели

- ✅ Аудиовыходы определяются через `/proc/asound/cards` (ALSA), а не
  PipeWire/PulseAudio — работает на headless systemd-сервисе без сессии
  (на RPi/trixie `pactl`/`wpctl` без сессии не подключаются). Выбор HDMI /
  3.5 мм применяется самим mpv через `--audio-device=alsa/plughw:CARD=…`;
  смена в панели перезапускает mpv с новым выходом.
- ✅ install.sh ставит `alsa-utils`, добавляет пользователя в группу `audio`;
  сервис получает `SupplementaryGroups=… audio`.
- ✅ Исправлен краш `'latin-1' codec can't encode…` при действиях панели с
  кириллицей в сообщении (перезапуск агента и др.): заголовок `Location`
  теперь процент-кодируется.

## 14. v0.9 — сохранность данных при обновлении

Проблема: данные лежали в именованном Docker-томе `signage-data`, который
удаляется командой `docker compose down -v` → при таком обновлении пропадали
экраны и загруженный контент.

- ✅ Хранение переведено на **bind-mount в папку `./data`** на хосте
  (`SIGNAGE_DATA_PATH`). Переживает пересборку, `down` и `down -v`
  (проверено); данные видны в ФС и легко бэкапятся.
- ✅ `scripts/update.sh` — безопасное обновление (бэкап + `up -d --build`,
  без `-v`); `scripts/backup.sh` / `scripts/restore.sh` — резервные копии
  (tar.gz, ротация 10 шт.).
- ✅ README: раздел про данные/обновление/бэкап и миграцию со старого
  именованного тома.
